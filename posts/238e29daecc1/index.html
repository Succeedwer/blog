Jedis连接Redis Cluster的java.net.SocketTimeoutException: connect timed out

环境

  • JDK 1.8
  • Redis 6.2.1
  • Redis Cluster 有6个节点，端口7001到7006，使用的内环地址127.0.0.1搭建。
  • 最新版本，2021年4月份发布的
  • 环境使用了ACLS，Access Conttrol Lists
  • Jedis 3.6.0
  • Redis Cluster 搭建环境： Ubuntu 19.10
  • RedisDesktopManager 可视化工具

问题再现

框架使用：

  • Spring-data-redis 2.5.0
  • Jedis 3.6.0

spring-redis.xml

使用了org.springframework.data.redis.connection.RedisPassword注入密码，节点Host皆为服务器地址192.168.1.102。用的测试类：

报错：

过程

从怀疑代码开始，问题解决永远先从自身找问题。

  • 配置问题 无
    参考spring-data-redis文档，并没有发现问题
  • 代码问题 无
    一句调用代码，不可能吧，spring启动也正常
  • 依赖版本问题 无
    降低Jedis版本，2.9.0 ，3.5.2
    结果 ：报错换了个，ClassNoFoundException: JedisClientConfig No Found
    2.9.0 还不支持ACL
    直接连接一个redis，没问题
  • Ubuntn 和 运行环境 防火墙问题 无
    端口都开放了
  • Redis 版本问题 无
    6.0 大版本出了快一年了，不存在不兼容，而且一直在使用Redis 6.2.1
    测试其他局域网内 用redis-cli 连接Redis Cluster，发现能连接上，使用正常。
    使用RedisDesktopManager ，依然没问题。
  • Jedis 有bug?
    从报错地方出发，发现HostAndPort是192.168.1.102 7001(7001-7006) 没问题，socket能成功连接。
    超时的都是127.0.0.1 7001 (7001 - 7006)。
    这不就是问题关键么？
    能通才奇怪呢，但是这个127.0.0.1从哪来的呢？
    跟踪报错，居然发现是从Jedis内的redis.client.jedis.Connection获取的HostAndPort 发现127.0.0.1居然是Redis返回的。
    • 猜想：
      • Jedis 开始socket连接192.168.1.102 7001并成功了，Redis 返回获集群里所有节点的IP127.0.0.1 7001 …
      • Jedis 开始连接返回各个节点的IP，连接127.0.0.1 7001失败，然后连接127.0.0.1 7002失败 …
      • Jedis 发现都失败了，开始连接配置中第二个节点IP192.168.1.102 7002，又拿到Redis返回的集群所有节点IP127.0.0.1 7001 …
        ···
      • 然后连接超过，配置中的maxRedirects，并且每次socket连接都超过超时时间2s，然后就报错java.net.SocketTimeoutException: connect timed out。
    • 提出问题：
      • 为什么redis 会返回127.0.0.1呢？ 和搭建Redis Cluster时，搭建集群使用回环地址127.0.0.1有关？
      • 如果有关，为什么使用其他工具(同一局域网内的RedisDesktopManager，redis-cli)能正常连接并使用？
    • 实验：
      • 重新搭建Redis Cluster，并使用服务器地址192.168.1.102的host，搭建。
    • 结果：
      • 代码运行成功，也能设置值，和拿到值。
    • 再一些问题：
      • 为什么同一局域网内的RedisDesktopManager，redis-cli能正常使用？
      • 这是Jedis的bug？都已经配置了节点ip，为什么还要从redis拿回节点ip?
      • 既然要去拿节点ip，为什么还要配置那么多个？
    • 再一些猜想：
      • 同一局域网内的RedisDesktopManager，redis-cli内部也许做了对回环地址的转换，就是当拿到的是127.0.0.1就转换该服务器的IP地址192.168.1.102；Jedis不会转换回环地址，是个耿直boy，如果转换，同一局域网内能连接上，统一服务器也能连接上。
      • 显然从某一节点，获取其他节点IP，从逻辑上来说，是一个很好机制。一开始获取全部节点IP，并进行连接放进连接池，需要使用时，直接使用即可，如果找不到一个可用连接，就会进行重试，超过重试次数，就报错。
      • 假如，某一个配置好的master节点A宕机了，那么它的slave节点中就会有一个晋升到master节点B，而这个新的节点B，并没有在我们的配置文件中，那么需要读写B节点的数据时，就会从其他节点重定向(Redirect)到新的节点B，Redis返回节点B的IP，那Jedis就能连接B节点，进行执行了。那么如果宕机的刚好就是配置中唯一的一个，连接都连接不上，何来获取其他节点IP。配置多个节点能够增加容错率，一个连不上，有其他可以进行连接。

问题解决

搭建Redis 集群使用了回环地址127.0.0.1，再加上Jedis 耿直boy，没有转换回环地址。需要使用服务器地址去搭建Redis集群。

总结

遇到socket连接超时，从以下几个方向考虑：
- spring配置问题，maxRedirects重连次数太少，超时时间设置太短 。
- 网络环境差，适当增加重连次数，增加超时时间，Jeids中socket默认超时时间2s。
- 防火墙关闭了端口，pingpong测试，以及查看运行环境和服务器的防火墙策略。
- 依赖版本过高，或过低，是否兼容Redis版本的新特性。Jedis3.6.0不支持jdk1.7。
- Redis Cluter 是否正常运行，以及是否用了回环地址搭建。
