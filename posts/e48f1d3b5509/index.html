面试题之Java基础

1. 问题：Vector和ArrayList区别？

  • 相同点
    • 都使用数组结构实现的。
  • 不同点
    • Vector是线程安全的，效率低。
    • ArrayList是线程不安全的，效率高。
    • Vector相对于ArrayList查询慢，因为线程安全。

2. 问题：ArrayList和LinkedList区别？

  • 相同点
    • 同属于Collection下的List接口的实现类。
    • 都是用于存储引用数据类型，其长度都能自动增长。
    • 都是线程不安全的。
  • 不同点
    • ArrayList是基于数组实现的，而LinkedList是基于链表实现的。
    • ArrayList
      • 查询快，通过下标索引查找目标索引，遍历下标索引即可。
      • 增删慢，在一般情况下在尾部添加元素快，但是总体不及LinkedList。
      • 修改快，修改之前需要进行查询，查询快，修改也快。
    • LinkedList
      • 增删快，实质就是改变前后节点的指针指向地址。
      • 查询慢，即需要遍历链表。
      • 修改慢，修改之前需要进行查询，所以慢。

3. 问题：ArrayList如何进行实现自动增长(扩容)？

  1. ArrayList扩容的实质是进行数组拷贝，将原本的数组元素拷贝到新的数组中。
  2. 每次进行扩容时，新数组的容量是原本的1.5倍。

    ps: 使用ArrayList应该避免数组扩容影响性能，可以创建时指定容量，以避免发生扩容。默认容量是10。

4. 问题：使用Araarys.asList()进行数组转集合，有什么需要注意的？

  1. 不能增加删除元素。
    Arrays.asList()得到的ArrayList并非日常使用的ArrayList，而是ArrayList中的静态内部类，该内部类继承于抽象类AbstractList，而AbstractList定义了add，remove，直接返回异常UnsupportedOperationException，而静态内部类并没有重写add，remove。所以只要使用了add，remove就导致UnsupportedOperationException。

  2. 需要转换的数组只能存储引用数据类型，不能存储基本数据类型，如：int。
    基本数据类型数组装换成集合，会将整个数组当作一个元素进行存储。

5. 问题： HashMap和Hashtable的区别？

  • 相同点：
    • 底层都是哈希算法，双列集合(数组+链表)。
  • 不同点：
    • HashMap是线程不安全的，效率高，JDK1.2版本后。
    • Hashtable是线程安全的，效率低，JDK.1.0版本后。
    • HashMap可以存储null键和null值，而Hashtable不行（equals方法需要非空对象）。

6. 问题：HashMap的原理？

  1. 数据结构
    HashMap是双列集合，内部是数组加链表(或红黑树)，1.7时只是数组加链表。
    1.8中，链表节点数大于8，链表就会转换成红黑树，当红黑树节点数少于6，红黑树就会转变成链表。
  2. 插入原理
    流程图如下

    a. 一开始对数组是否为空进行判断，为空就初始化数组。
    b. 通过(n-1)&hash计算存储位置，n为数组长度，hash为录入键值对的key的hash值。
    c. 得出指定位置，判断指定位置是否有数据，如果没有，直接创建节点存储数据。
    d. 如果有，说明发生了哈希冲突，继续判断新数据和旧数据的key值，hash值是否相同，如果相同，新值替换旧值并返回。
    e. 如果不相同，判断当前节点类型是否为红黑树节点，如果是，遍历红黑树，判断key值，hash值是否与存在的节点的key值，hash值相同；如果相同，替换旧值并返回，如果不相同，创建树节点并加入红黑树。
    f. 如果不是红黑树节点，遍历当前链表，判断key值，hash值是否与存在的节点的key值，hash值相同；如果相同，替换旧值并替换，如果不相同，创建普通节点并加入链表。
    g. 插入完成后，判断当前总节点数是否大于阈值，如果大于，对数组进行扩容到n*2的大小，即原本数组容量的2倍。

7. 问题：1.8后对HashMap进行怎么样的优化，为什么？

优化：

  1. 数组+链表改成数组+链表(或红黑树)。
  2. 链表的插入方式由头插法改成尾插法。即插入时，数组上存储位置已有节点，1.7是将新节点放入到数组中，数组中原本节点变成新节点的后继节点。1.8是遍历链表，将新节点放在链表最后。
  3. 扩容时，1.7是需要对所有节点进行重新hash定位；1.8则不需要继续重新hash，经过简单的逻辑运算，重新定位，索引不变或索引+旧容量大小。
  4. 插入时，1.7需要先判断是否需要扩容，再插入；1.8先进行插入，插入完成后再进行扩容判断。
  5. 1.8对hash函数(扰动函数)做了优化，只进行了一次异或和移位，1.7做了4次移位和异或。

原因：

  1. 链表改成红黑树，防止发生hash冲突时，链表长度过长，可以将链表的时间复杂度O(n)降低到O(logn)。
  2. 1.7头插法扩容时，会导致链表反转，多线程情况下还会产生环的数据结构。而先尾插法插入后扩容则不会。
  3. 1.8对hash进行了改进，扩容时，计算数组的位置的掩码仅仅只是高位多了一个1。(n-1)&hash下，扩容，即(2n-1)&hash，即(2n-1)比(n-1)的高位多一个1，而1与hash进行&逻辑运算，无非是和0和1，那么得到的结果也就是原本位置和原本位置+原本数组的容量两种结果。
  4. 因为1.7先扩容后插入，会导致链表反转，甚至产生环;而先插入，再扩容，则能避免。
  5. 移位和异或（扰动）做多了，边际效用不大，降低次数能提高效率。

  1. 尽可能降低hash碰撞，越均匀分散越好。
  2. 因为高频操作，所以算法尽可能高效，故而使用位运算。

为什么呢？

已经知道对key进行hashCode()返回的结果是int类型散列值，int值范围为-232 ~ 231-1，显然40多亿映射空间，很难出现hash碰撞，但是内存也放不下，所以要对它进行取余运算，即&运算，即(n-1)&hash，得到的结果作为下标才能去访问数组。

HashMap的数组长度必定是2的整数幂，因此进行&与运算时，就能截取出hash的低位值。问题来了，就算散列值分布再分散，只截取后几位，配置可谓是相当严重。如何解决呢？

扰动函数(hash函数)，将key.hashCode()的值无符号向右移动16位再异或，相当于将高半区和低半区做异或^运算。那么就能混合哈希码的高位和地位，以此来加大低位的随机性，而混合后的低位掺杂了高位的部分特征，这样高位的信息也被变相保留下来。

10. 问题：HashMap是线程安全的么？

HashMap并非线程安全的，在多线程情况下，1.7会产生死循环，数据丢失以及数据覆盖的问题。1.8中会有数据覆盖的问题。

解决以下几种方法：

  1. Hashtable
    Hashtable是直接在操作方法上加synchronized关键字，锁住整个数组，粒度大，并发量低。
  2. Collection.synchronizedMap
    Collection.synchronizedMap是使用Collections集合工具的内部类，通过传入Map封装出一个SynchronizedMap对象，内部定义一个对象锁，方法内通过对象锁实现。
  3. ConcurrentHashMap
    ConcurrentHashMap使用分段锁，降低锁粒度，让并发度大大提升。

11. 问题：ConcurrentHashMap的分段锁实现原理？

ConcurrentHashMap成员变量使用volatile关键字修饰，免除了指令重排序，同时保证内存可见性，另外使用CAS操作和synchronized结合实现赋值操作，多线程操作只会锁住当前操作索引的节点。

如：线程A锁住A节点锁住链表，线程B锁住B节点所在链表，操作互不干涉。

  1. ConcurrentHashMap无论读操作和写操作都具有很高的性能。
  2. 读操作几乎不需要加锁，而写操作时通过锁分段技术只对所操作的段加锁而不影响其他段的操作。
  3. ConcurrentHashMap能够最大支持16个线程执行并发操作，以及任意线程的读操作。
  4. 1.7 中使用分段锁ReentrantLock + Segment，相当于一个HashMap分成多段，每一段有一把锁，这样支持多线程访问。锁粒度：基于Segment，包含多个HashEntry。
  5. 1.8中使用CAS + synchronized，锁粒度大大降低，为Node节点。

12. 问题：List、Set、Map三个接口的区别以及其常用实现类？

  1. List、Set、Map都是Collection结构中的接口类。
  2. List、Set是单列集合，Map是双列集合。
  3. List、Set是基于Collection接口的，而Map是Collection容器内的两大基本接口之一(Collection,Map)。
  4. List是有序的，可重复的，Set是无序的，不可重复的。
  5. List常用实现类为ArrayList，LinkedList和Vector。
  6. Set常用实现类为HashSet和TreeSet。
  7. Map常用实现类为HashMap，TreeMap以及HashMap的子类LinkedHashMap。

13. 问题：抽象类与接口的区别？

  1. 抽象类是用于捕捉子类的通用特征的，而接口则是抽象方法的结合。
  2. 抽象类不能实例化，只能作为子类的超类，是被用于创建继承层级里子类的模板，而接口只是一种形式，更像一种约定，无法做任何事。
  3. 抽象类使用abstract关键字修饰，子类使用extends关键字来继承抽象类；而接口使用interface关键字修饰，实现类使用implements关键字实现接口。
  4. 抽象类除了不能实例化抽象类之外，和普通Java类没有任何区别，能够拥有成员变量，构造器，方法等。
  5. 接口只能有常量(public static final修饰(可省略))，抽象方法，默认方法(1.8)，静态方法(1.8)，私有方法（1.9），私有静态方法（1.9）。没有构造器。
  6. 一个类只能继承一个抽象类，但能实现多个接口。

14. 问题：Java的代理是怎么样的？

代理涉及到代理模式，什么是代理模式呢？

代理模式：A给B提供服务(协议)，并由A来控制对B的访问，称作A对B进行代理。代理模式是一种结构型设计模式。

代理模式角色分为三种：

Subject（抽象主题角色）：定义代理类和真实主题的公共对外方法，也是代理类代理真实主题的方法。如：协议（抽象层）

RealSubject（真实主题角色）：真正实现业务逻辑的类。

Proxy（代理主题角色）: 用来代理和封装真实主题。

  1. 代理从字节码创建时机来说，分为静态代理和动态代理。
  • 静态代理是程序运行之前就已经存在了字节码文件，代理关系已经确定了。
  • 动态代理则是程序在运行期间有JVM根据反射等机制动态生成字节码，所以在运行之前不会存在代理类的字节码文件。
  1. 静态代理的优缺点：
    • 优点：可以很好避免目标对象的业务逻辑对外暴露，从而提高安全性；也可以在不影响目标对象的情况下，进行功能增强。
    • 缺点：不同接口需要不同的代理对象实现，导致代码冗余。
  2. 动态代理根据实现方式分为：
    • JDK动态代理
    • CGLIB动态代理
  3. JDK动态代理：基于Java反射机制来实现的，即需要目标对象必须实现某一接口。
    CGLib动态代理：基于ASM机制实现，通过生成目标对象的子类作为代理对象。
  4. JDK动态代理
    • 优点：解决了静态代理中代理对象的冗余问题；最小化依赖关系，简化开发和维护成本；JDK自身的支持，比CGLIB更可靠；
    • 缺点：JDK动态代理是基于接口设计实现的，必须依赖接口，否则会报异常ClassCastException。
  5. CGLIB代理
    • 优点：不需要用接口就能实现动态代理，采用ASM字节码增强技术。
    • 缺点：创建代理对象的消耗比JDK动态代理要大，性能较与JDK要稍微差点。
  6. 选择：
    • 对于无需频繁创建的的对象，如单例，使用CGLIB合适，反之使用JDK方式比较好。
    • 由于CGLIB是采用动态创建子类的方法，对于final修饰的方法，无法进行代理。

15. 问题：堆和栈的区别？

  1. 数据结构，堆和栈是指数据结构。
    • 堆在数据结构中是满足父子节点大小关系的完全二叉树。
    • 栈在数据结构中呈线性结构，具有先进后出的特点，如一个桶，先进会被压栈。
  2. 操作系统中，堆和栈是指内存空间，更准确的叫法，应该是堆区（heap）和栈区（stack）。
    • 申请方式
      • 堆是需要程序员申请的，并指明大小的，手动申请，手动释放，如C里面的malloc和free，不过这一般由操作系统去回收。
      • 栈是由编译器自动申请和自动释放的的。
    • 申请大小限制
      • 堆是向高地址扩展的不连续的内存区域。由于系统是用链表去存储空闲的内存地址，所有堆就是不连续的内存区域，而且链表遍历是从低地址向高地址的。堆的大小受限于系统的有效虚拟内存空间，因此堆获得的空间比较灵活，也比较大。
      • 栈是向低地址扩展的一块连续的内存区域。栈顶地址和大小事先由系统规定好的，能获得的栈空间有限。
    • 申请效率
      • 堆由程序员分配，速度较慢，容易产生内存碎片。
      • 栈由系统分配，速度快，但程序员无法控制。
    • 存储内容
      • 堆的存储内存由程序员决定，如对象等实例。
      • 栈主要存储局部变量,如 int a; 在函数调用时，首先进栈的是主函数中函数调用后的下一条指令的地址，然后是函数的各个参数，在C中，参数是从右到左入栈的，再到局部变量，当函数调用结束后，局部变量出栈，然后是参数，然后栈顶指针指向的是最开始的存的地址，即主函数中的下一条指令。

18. 问题：Java常用的设计模式有哪些？

Java设计模式共有23种设计模式。

常用的有8种

  • 单例模式(Singleton)：某个类只能生成一个实例，该类提供了一个全局访问点供外部获取该实例，其拓展是有限多例模式。
  • 建造者模式(Builder)：将一个复杂对象分解成多个相对简单的部分，然后根据不同需要分别创建它们，最后构建成该复杂对象。
  • 抽象工厂模式(Abstract Factory)：提供一个创建产品族的接口，其每个子类可以生产一系列相关的产品。
  • 工厂方法模式(Factory Method)：定义一个用于创建产品的接口，由子类决定生产什么产品。
  • 观察者模式(Observer)：多个对象间存在一对多关系，当一个对象发生改变时，把这种改变通知给其他多个对象，从而影响其他对象的行为。
  • 模板方法模式(Template Method)：定义一个操作中的算法骨架，而将算法的一些步骤延迟到子类中，使得子类可以不改变该算法结构的情况下重定义该算法的某些特定步骤。
  • 装饰者模式(Decorator)：动态的给对象增加一些职责，即增加其额外的功能。
  • 代理模式(Proxy)：为某对象提供一种代理以控制对该对象的访问。即客户端通过代理间接地访问该对象，从而限制、增强或修改该对象的一些特性。

19. 问题：设计模式的六大原则有哪些？

  • 单一职责原则(Single Responsibility Principle)
    • 一个类只负责一个功能领域中的相应职责，或者可以定义为：就一个类而言，应该只有一个引起它变化的原因。
  • 开闭原则(Open-Closed Principle, OCP)
    • 一个软件实体应当对扩展开放，对修改关闭。即软件实体应尽量在不修改原有代码的情况下进行扩展。
  • 里氏代换原则(Liskov Substitution Principle, LSP)
    • 所有引用基类（父类）的地方必须能透明地使用其子类的对象。
  • 依赖倒置原则（Dependence Inversion Principle，DIP）
    • 高层模块不应该依赖低层模块，两者都应该依赖其抽象；抽象不应该依赖细节，细节应该依赖抽象，其核心思想是：要面向接口编程，不要面向实现编程。
  • 接口隔离原则(Interface Segregation Principle, ISP)
    • 使用多个专门的接口，而不使用单一的总接口，即客户端不应该依赖那些它不需要的接口。
  • 迪米特法则(Law of Demeter, LoD)也叫最少知识原则（Least Knowledge Principle，LKP)
    • 一个软件实体应当尽可能少地与其他实体发生相互作用。
    • 如果两个软件实体无须直接通信，那么就不应当发生直接的相互调用，可以通过第三方转发该调用。其目的是降低类之间的耦合度，提高模块的相对独立性。

20. 问题：讲一下单例模式？

单例模式有5种写法，饿汉式，懒汉式，双重检查锁，静态内部类法，枚举法(推荐)。

四大原则：

  1. 构造器私有
  2. 返回值必须是以静态方法或枚举返回。
  3. 保证线程安全，只允许一个实例。
  4. 序列化和反序列化安全
    延迟加载，并不一定要求。
  1. 饿汉式（静态常量，或静态代码块）
    静态常量 静态代码块 优点: 写法简单，类装载时候就完成实例化了，避免线程同步的问题。
    缺点: 在类装载时就已经实例化了，未能达到懒加载(Lazy Load)效果，可能会造成内存浪费。
  2. 懒汉式(线程安全，同步代码块) 使用关键字volatile 进行限制，保证其对所有线程可见性，并且禁止对其进行指令重排序
    缺点： 效率仍然低，每次调用getInstance，会在synchronized进行排队
  3. 双重检查锁 进行多一层判断instance是否为空，大大提高并发，提高效率。
    volatile 有两层语义：可见性 和禁止指令重排序
    可见性：在一个线程中对该变量的修改会马上由工作内存（Work Memory）写回主内存（Main Memory），所以会马上反应在其它线程的读取操作中。
    工作内存是线程独占的，而主内存是线程共享的。工作内存和主内存可以近似理解为实际电脑中的高速缓存和主存
    禁止指令重排序：由于编译器优化，在实际执行的时候可能与我们编写的顺序不同。编译器只保证程序执行结果与源代码相同，却不保证实际指令的顺序与源代码相同。
    这在单线程看起来没什么问题，然而一旦引入多线程，这种乱序就可能导致严重问题。volatile 关键字就可以从语义上解决这个问题。
    在jdk1.5后，volatile才能真正实现禁止指令重排序，1.5之前不能正确工作。
    所以在jdk1.5之前，双重检查锁是不安全的。
  4. 静态内部类法 静态内部类不会在outer类加载中被加载，只有等到调用静态内部类时，才会加载，起到懒加载的作用。
    同时静态内部类只会加载一次，也保证了线程安全。
  5. 枚举法 使用枚举除了线程安全和防止反射强行调用构造器之外，还提供了自动序列化机制，防止反序列化的时候创建新的对象。
    因此，Effective Java 推荐尽可能地使用枚举来实现单例。
    注意： 除了枚举法之外，都需要额外的工作 (Serializable、transient、readResolve()) 来实现序列化，否则每次反序列化一个序列化对象实例时都会创建一个新的实例。而且还可以使用反射来破坏单例，需要对构造器进行修改，使得创建第二个实例时抛异常。而枚举法除了线程安全，防止反射强行调用构造器，还提供了自动序列化机制。

21. 问题：排序算法有哪些？

  1. 冒泡算法 时间复杂度O(n2) 空间复杂度O(1)

  2. 插入排序 时间复杂度O(n2) 空间复杂度O(1)

  3. 快速排序 时间复杂度O(nlogn) 空间复杂度O(1)

  4. 选择排序 时间复杂度O(n2) 空间复杂度O(1)
