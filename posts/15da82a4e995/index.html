6.workflow模式

介绍

workflow模式是dtm首创模式。简单来说，就是混合使用XA、SAGA、TCC事务模式，还可以混用Http、GRPC、本地操作。

该模式有着极大灵活性，给予开发人员极大的定制内容。

使用

简单使用

以saga模式简述workflow事务的实现流程。

需要使用混合协议进行事务，有三个步骤。

  • 初始化SDK

    • HTTP

      func InitHTTP(httpDtm string, callback string) 初始化HTTP

      • httpDtm： 指定dtm服务http地址
      • callback：回滚地址

    • GRPC

      func InitGrpc(grpcDtm string, clientHost string, grpcServer *grpc.Server)初始化GRPC

      • grpcDtm：指定dtm服务grpc地址
      • clientHost：指定业务服务器地址
      • grpcServer：grpc服务

      workflow需要从业务服务器地址 +grpcServer上接收dtm服务器的回调。

      注意：初始化grpc必须要在server.serve()之前，因为在workflow.InitGrpc()将注册workflowServer到grpc server。

  • 注册workflow

    func Register(name string, handler WfFunc, custom ...func(wf *Workflow)) error注册workflow函数

    • name：workflow事务名称
    • handler：全局事务操作函数
    • custom：自定义操作函数

  • 执行workflow

    func ExecuteCtx(ctx context.Context, name string, gid string, data []byte) ([]byte, error)

    • ctx：上下文
    • name：workflow事务名称
    • gid：全局事务id
    • data：子事务body参数
    • 返回值为nil/ErrFailure，全局事务已成功/已回滚。
    • 返回值为其他值时，dtm服务器后续会回调该工作流任务进行重试。

saga

http

  • OnRollback: 设置回滚
  • wf.NewRequest().SetBody(req).Post(server.HttpUrl + "/TransOutRevert") ：创建事务分支，使用http协议，这里是补偿操作。
  • .NewRequest().SetBody(req).Post(server.HttpUrl + "/TransOut")：使用http协议，正向操作。

grpc

  • wf.NewBranch().OnRollback()： 设置回滚，然后在函数参数中，调用回滚操作。
  • server.RMClient.TransIn(): 在设置回滚之后，直接调用rpc。

注意：参数的序列化和反序列化要使用protobuf。

tcc

http

  • OnRollback: 设置回滚，cancel阶段操作
  • OnCommit: confirm阶段操作
  • .NewRequest().SetBody(req).Post(server.HttpUrl + "/TransOut"): try阶段操作。

grpc

在编写好rollback/confirm后，直接调用rpc即可。

xa

本地操作

目前在文档中，只找到本地操作的例子。

  • .DoXa(data.DBConf, func(db *sql.DB) ([]byte, error)): 开启xa模式事务
    • data.DBConf： 数据库配置
    • func(db *sql.DB) ([]byte, error): 实际操作

多种模式混合使用

该例子，用了2种事务模式，saga和tcc，用了grpc、http、本地操作协议。

自定义返回错误

事务完成通知

OnFinish()：当该事务分支完成时，会调用该回调函数，其中isRollback可以作为是否回滚的依据。

理论

分支操作结果

  • 成功：分支操作返回HTTP-200/grpc-nil
  • 业务失败：分支操作返回HTTP-409/gRPC-Aborted，不再重试，全局事务进行回滚。
  • 进行中：分支操作返回HTTP-425/gRPC-FailPrecondition，表示事务正在正常进行中，dtm会以固定间隔重试。
  • 未知错误：分支操作返回其它结果，dtm会以指数退避算法重试。

如果现有服务返回结果与上述结果不一致，那么可以通过workflow.Options.HTTPResp2DtmError/GRPCError2DtmError来进行定制结果。

saga的补偿操作，tcc的confirm/cancel操作，根据它们的协议，是不允许返回业务上的失败的，即使返回错误，也不会进行重试。

在设计开发中，要注意避免返回错误，否则在运维上要做通知并手动补偿操作。

事务完成通知

如果业务需要获得事务完成的通知，可以在事务分支上设置OnFinish回调来实现。

有一个地方需要注意，收到OnFinish回调时，dtm服务器上，这个事务的状态还未修改为最终状态，因此如果混合使用事务完成通知和查询全局事务结果，那么两者的结果可能不一致，建议用户只使用其中一种方式，而不要混合使用。

workflow如何保证分布式事务的数据一致性？

当业务进程出现crash等问题时，dtm服务器会发现该workflow全局事务超时未完成，则dtm会采用指数回避策略进行重试。

当重试请求到业务服务时，SDK会从dtm服务器读取全局事务的进度，对于已完成的分支，会将之前保存的结果，通过grpc/http等拦截器，直接返回分支结果。最终workflow会顺利完成。

同样的，workflow事务函数也是幂等。

workflow如何实现幂等？

当crash发生，workflow会进行重试，因此要求各个分支事务支持幂等。

一般情况下，可以采用数据库insert ignore "unique-key"实现幂等。

如果插入失败，说明数据已经操作了，就直接忽略返回。

如果插入成功，说明时首次操作，继续进行业务操作。

如果业务本身支持幂等，那么直接操作即可。

如果没有支持幂等，那么可以使用dtm的BranchBarrier辅助类，基于上述unique-key原理，实现幂等操作。

支持的数据库：Mysql、Mongo、Redis等。

两个典型非幂等案例：

  • 超时回滚
    • 事务操作耗时长，如果超过事务超时时间，返回失败，进行回滚；那么这就不是幂等，并发情况下，一个操作成功了，一个操作失败回滚。
  • 达到重试上限后回滚：并发情况下，一个重试后成功了，一个超出重试上限回滚。

workflow暂不支持超时回滚和达到重试上限后回滚。

混合使用事务

workflow模式下，混用多种事务模式，能够根据分支事务的特性选择子模式。

  • xa：如果业务不需要竞争行锁，且对并发要求低，那么就可以采用xa。
    • Commit/Rollback由数据库自动完成，适合创建订单。
    • 不适合扣减库存，因为涉及到商品订单竞争行锁。
  • saga：不适合xa的普通业务都适合saga。
    • 长事务，如订票，需要第三方确认的，时间跨度长的业务。
  • tcc：适合高一致性的业务。
    • 转账

协议支持

GRPC

  • 初始化workflow.InitGrpc

  • 通过grpc的拦截器，自动将调用结果保存到dtm服务器。并在workflow再次执行时，自动将dtm服务器保存的结果，直接返回给调用者。

    如：_, err = RMClient.TransIn(wf.Context, req)

HTTP

  • 初始化workflow.InitHttp

  • 通过http的拦截器，自动将调用结果保存到dtm服务器。并在workflow再次执行时，自动将dtm服务器保存的结果，直接返回给调用者。

    ·如：_, err := wf.NewBranch().NewRequest().SetBody(req).Post(server.HttpUrl + "/TransOut")

本地操作

示例如下：

wf.Do(fn func(bb *dtmcli.BranchBarrier) ([]byte, error)) 在参数fn函数中，进行本地事务即可，还可以配合子事务屏障使用。func (bb *BranchBarrier) CallWithDB(db *sql.DB, busiCall BarrierBusiFunc) error

混用协议

grpc/http/本地操作，也可以混用配合事务分支。

可以简单得将遗留系统接入分布式事务。
