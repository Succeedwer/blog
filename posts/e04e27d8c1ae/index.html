1.缓存一致性之概念

引言

这里所说的缓存一致性，是指数据库和缓存数据一致性（如mysql和redis）。

为什么要做缓存？

应用程序对同一个数据进行重复访问时，而使用缓存可以提高性能。

在绝大部分情况下，是考虑到减少数据库压力，同时从缓存读取，性能一般较从数据库读取高。

如何做缓存？

从数据库读取出数据，保存到缓存中，并设置过期时间。

为什么缓存必须要有过期时间？

对于系统来说，缓存命中率越高，系统性能越好。

如果缓存没有设置过期时间，命中率也低的时候，那么浪费了存储空间，还浪费了系统计算能力。

如果缓存命中率高，那么在每次命中时刷新过期时间，就能保证缓存一直存在，从而保证了缓存的命中率，提高系统性能。

同时，设置过期时间也是一个保证数据最终一致性的手段，当数据库和缓存数据不一致时，过期时间到了，直接访问数据库，也保证了数据的最终一致性。

数据库和缓存数据能做到完全一致性？

这种想法时不切现实的，至少目前业界未看到完美的方案。

如mysql和redis为例，两个外置系统在保存数据到缓存和保存到数据库必定存在时间差，则必然出现数据不一致的时间窗口。

通常情况下，窗口不大，影响就不大，如果当两个操作中间发生宕机crash，就有可能发生完全其中一个操作，另一个未完成，那么就会导致数据出现长时间不一致。

保存数据到缓存和保存到数据库这两个操作原子性也是尤为重要的。

这里将缓存一致性的强弱程度分为3种类型。

  • 简单一致性：允许数据库和缓存存在较短的缓存不一致的窗口期，读多写少，对性能影响低。
  • 强一致性：数据库和缓存的数据要求强一致性，对性能影响较高。
  • 最终一致性：数据库和缓存的数据在应用面前都是相同的，数据最终完全一致性，对性能影响最高。

缓存外模式（Cache Aside Pattern）

什么是Cache Aside Pattern？

其实这是一个保证数据库和缓存数据一致性的经典方案[Cache Aside Pattern](Cache-Aside pattern - Azure Architecture Center | Microsoft Learn)。

读缓存流程：

  1. 确定数据是否存在缓存中。
  2. 如果数据不存在缓存中，从数据库读取。（如果存在，直接读取。）
  3. 当数据不存在缓存中并从数据库读取后，将数据存储到缓存中。

写缓存流程：

  1. 先更新数据库，再删除缓存。

这种设计很简单，但是在写缓存流程中，为什么一定要先更新数据库，再删除缓存？

写缓存顺序

枚举一下写缓存的几种可能：

  • 先更新缓存，再更新数据库。
  • 先更新数据库，再更新缓存。
  • 先删除缓存，再更新数据库。
  • 先更新数据库，再删除缓存。

每种可能都有两个操作，这里都是在这两个操作非原子性情况下讨论。

先更新缓存，再更新数据库

更新缓存成功，更新数据库失败(如负责数据库更新的服务器crash)，那么数据库存储就是旧数据。

一旦缓存过期失效了，那么新数据就会丢失，这显然是不允许的。

方案先更新缓存，再更新数据库 NO!

先更新数据库，再更新缓存

在并发情况下，假设存在两个服务A、B同时更新同一个数据，因为网络情况，无法保证其顺序，那么就会这种情况：

  1. A更新了数据库，数据为a。
  2. B更新了数据库，数据为b。
  3. B更新了缓存，数据为b。
  4. A更新了缓存，数据为a。

那么最终数据库存储了b，而缓存存储了a。那么就会导致缓存和数据库的数据不一致。

再者，每次更新数据库就要更新缓存，那么如果该缓存的访问率低，就会造成缓存存储空间的浪费，也会造成数据的无效计算浪费机器性能。

方案先更新数据库，再更新缓存 NO!

先删除缓存，再更新数据库

同样的，在并发情况下，对缓存和数据库的操作，无法保证其顺序，就会导致数据不一致。

如服务A写操作，服务B读操作，就有可能出现一下执行顺序：

  1. A删除了缓存，数据为old。
  2. B读取缓存，发现缓存不存在，从数据库读取数据old。
  3. A更新数据库，数据为new。
  4. B写入缓存，数据为old。

那么在这种情况下，只要数据缓存未失效，就会一直读取旧数据。

在数据库读写分离的架构下，因为主从同步也是要一定的时间成本，这种情况出现更加频繁。

方案先删除缓存，再更新数据库 NO!

先更新数据库，再删除缓存

这种方案也是Cache Aside Pattern设计中的写操作，更是业务中常用的方案。

但也会出现问题，同样的服务A读操作，服务B写操作，就有可能出现一下执行顺序：

  1. 缓存失效。
  2. A读取数据库，得到数据d。
  3. B更新数据库，数据为D。
  4. B删除缓存
  5. A写入缓存，数据为d。

那么就会出现数据库和缓存数据不一致，当然发生这种情况要满足一下条件：

  • 在读取数据时，缓存刚好失效，并且有一个并发的写数据
  • 数据库的写操作比读操作慢（数据库写操作上锁，普通读操作不会上锁）
  • 一般业务都是读多写少

所以缓存中出现脏数据的情况可能性不高。

当然在读写分离的数据库架构中，同样有可能出现问题：

  1. A更新主库，数据为new。
  2. A删除缓存数据old。
  3. B查询缓存，发现不存在，查询从库得到old。
  4. 主从同步完成
  5. B写入缓存，数据为old。

当然啦，这种情况仅出现数据库主从同步比较慢的时候，就会出现数据不一致。

为什么仍然采用这个方案呢？

因为在持久化角度来看，应该以数据库的数据为主，所以先更新数据库，就算缓存出现crash，数据依然存在且准确。

当然，为了保证更新数据库后，及时删除缓存，除了缓存有过期时间这终极方案，也可以对删除缓存操作进行重试。如消息队列等。

如果更新数据库crash了，那么数据库和缓存中的数据仍为老数据，数据库和缓存数据仍然一致。
