1.rabbitmq之使用

前言

本文为amqp091-go的一些简单使用。

源码：go-rabbitmq · 疯狂の小托马斯/go-learn - 码云 - 开源中国

总体来说，要完成订阅/发布的流程，要进行以下步骤：

  • 连接RabbitMQ。
  • 打开通道channel。
  • 声明交换机。
  • 声明队列。
  • 绑定队列到交换机
  • 订阅/发布。

环境

  • Ubuntu 22.04.1 LTS
  • Docker version 27.3.1, build ce12230
  • RabbitMQ 4.0.2
  • amqp091-go v1.10.0

简单的pubsub

  1. 连接到RabbitMQ，获取connection。

  2. 打开通道channel。

  3. 声明队列，配置队列特性。

  4. 发布消息。

    以下是amqp.Connection.PublishWithContext的主要参数解释：

    • context：上下文。

    • exchange：指定消息要发送到的交换机名称。

    • key：路由消息的值，决定消息将被发送到哪些队列。

    • mandatory：指的是是否必须将消息按路由规则分配到至少一个队列。

      • mandatory = true
        • 在生产者发送消息时，RabbitMQ会检查是否有队列绑定到指定的交换机，并且消息能够根据路由规则正确分配到队列。
        • 如果消息没有被分配到队列，那么RabbitMQ将通过一个特定的回调机制将消息返回到生产者。
      • mandatory = false
        • 如果消息没有被分配到队列，那么RabbitMQ将丢弃该消息，生产者不会收到任何反馈。

    • immediate：指的是消息是否立即投递。

      • immediate = true

    • 消息被发送到消费者时，消费者在消息发布时必须处于准备接收的状态。

      • 如果消费者没能立即接收该消息，RabbitMQ会丢弃该消息，不会放在队列中。
      • 多用于需要实时处理消息的场景，确保消息不会被放入队列，而是直接发送给“待接收”的消费者。

    • immediate = false

      • 即使没有消费者立即接收消息，消息也能被正常处理，RabbitMQ将其放入队列中，等待消费者接收。

    • msg：消息，包含多个主要可选参数：

      • ContentType：消息的内容类型（如text/plain）
      • ContentEncoding：消息编码格式（如utf-8）。
      • Body：消息的二进制。
      • headers：附加的自定义头部消息。
      • DeliveryMode：消息投递模式，0和1表示临时，2表示持续
      • Priority：优先级，0到9
      • CorrelationId：请求和响应关联的id
      • ReplyTo：指定接收回复的队列名称。

  5. 订阅消息。

    以下是channel.Consume函数的主要参数：

    • queue：接收消息的队列名称。

    • consumer：消费者标签，默认为空字符“”，RabbitMQ会自动生成唯一消费标签。

    • autoAck：是否自动确认消息，为false时，需要手动确认。

    • exclusive：消费者的排他性。

      • exclusive = true：
        • 消费者被标记为排他性消费者，即该消费者只能与声明它的连接使用。
        • 一旦该消费者关闭或断开连接，RabbitMQ会自动删除其绑定的队列。
        • 通常用于临时队列，确保其他消费者无法访问该队列。
        • 也可以用于一次性处理的工作，或在特定的连接中需要进行隔离任务。
      • exclusive = false：
        • 消费者可以共享队列，即其他消费者也可以连接到同一个队列。
        • 该消费者关闭或断开连接，队列将继续存在，其他消费者可以继续消费。

    • noLocal：无本地标志，为true时，本地连接的生产者发送的消息不会被该消费者接收。这在多个连接时可以防止循环消费。

    • noWait：不等待标志。为true时，消费者声明请求不会等待RabbitMQ的响应，适合希望尽快返回的情况；为false时，则会等待确认。

    • args：附加参数，用于传递队列的额外配置（如特定的队列属性）。

交换机的几种类型

RabbitMQ自带的几种交换机：

  • Direct：根据路由规则，直接转发。
  • Fanout: 广播消息到所有绑定的队列。
  • Topic: 根据消息主题分发到队列。
  • Headers：根据消息头进行转发。

Direct Exchange

RabbitMQ的默认交换机为名为空字符串的直接交换机，可以直接声明队列，如果没有显式进行绑定，那么该队列就被自动绑定到该Direct交换机。

所以，就算不声明交换机，不进行绑定，只声明队列，并在队列中设置路由key，也可以将消息发送到该队列。

在发布消息时，路由key为空字符串时，表示消息将被发送到与交换机绑定的所以队列。

当未声明交换机或交换机类型为空字符串时，且发布消息时，路由key也是空字符串，那么可以理解为广播或无条件发送，即所有绑定了该交换机的队列都会接收到这条消息。

但是，在实际生产环境中，通常需要进行声明交换机，声明队列，指定路由key，并绑定队列。

主题模式，交换机根据消息的主题分发消息。

主题格式通常有单词、通配符（可选）（*和#）、分隔符.组成。如foo.bar.*，foo.bar.one。

通常主题的通配符有*和#。

  • *****：表示单个单词，如foo.bar.*、foo.*.bar能够分别匹配到foo.bar.one、foo.one.bar
  • **#**：表示多个单词，如foo.bar.#，能够匹配到foo.bar.one.two

QoS

QoS（Quality of Service）在RabbitMQ中 ，用于控制消息的消费速率，以确保消息处理的可靠性和稳定性。

主要通过以下几个方面体现：

  • 预取计数（prefetch count）：通过设置预取计数，消费者可以限制在处理未确认消息之前，可以从队列中获取的消息数量，防止消费者因一次接收过多消息而导致负载过重。
  • 消息确认：消费者可以在成功处理消息后显式确认消息（delivery.Ack），确保消息不会在处理失败后被丢失。
  • 优先级：指定消息优先级，使得优先级高得消息可以被优先处理。

函数qos的解释如下：

  • prefetchCount：定义了消费者在处理前可以接收的消息数量。设置为 1 时，消费者会在处理完当前消息后才会接收下一个消息。

  • prefetchSize：定义了消费者在处理消息时可以接收的总字节数。0表示无限制，>0通常只在需要严格控制内存时使用。

  • global：设置为true表示该QoS设置将适用于所有消费者；如果为false，则只适用于当前消费者。

如果在定义consumer时，使用了noAck，那么QoS的配置将失效。
