3.数据容器

数组和切片

1. 数组和切片有何异同?

切片本质是对数组的封装，描述一个数组的片段。

  • 同

    无论切片还是数组，都可以通过下标来访问单个元素。

  • 异

    • 数组是定长的，初始化定义后无法修改；切片可以动态扩容，切片的类型与长度无关。
    • 数组是一片连续内存；切片实际是一个结构体，包含三个字段：长度、容量、底层数组。

注意！底层数组可以被多个切片同时指向，因此对一个切片的元素进行操作可能会影响到其它切片。

2. 切片如何被截取？

截取是常见的创建slice的方法，在数组或已有的slice直接截取。

基于已有slice创建新slice对象，被称为reslice。新老slice共用底层数组，新老slice对底层数组的操作都会影响到彼此。

基于数组创建的新slice，对数组或slice的操作都会影响到彼此。

值得注意的是，当使用append操作使得新slice或老slice底层数组扩容，将数据进行拷贝移动到新位置，那么两者将不会互相影响。

截取方式:

low是最低索引值，为闭区间，包含当前索引的元素。

high是最高索引值，为开区间，不包含当前索引的元素。[low, high)

max是最大容量的索引，为开区间，而最大容量只能到索引max-1处。如果不写，默认为到数组结尾len(data)-1，即容量为max-low = 8。

注意，low <= high <= max，且high和max必须要在老数组或老slice的容量（cap）范围内。

3. 切片的容量是怎样增长的？

扩容的起因一般是向slice追加元素（append），由于容量不足，才会引起扩容。

go语言编译器不允许调用了append函数后不使用返回值！

append函数向slice追加元素，实际上是往底层数组对应的位置放置要追加的元素。

由于数组长度是固定的，所以如果数组已经满了，那么slice就整体迁移到新位置，并且底层数组的长度也会增加。而且扩容后的底层数组会预留一定的长度，以便应对再次发生的append函数。

那么扩容的规律是怎么样的呢？

并非2倍，1.25倍，而是与内存分配策略相关。

在底层代码中，slice调用/src/runtime/slice.go中的growslice函数进行扩容。

进行内存对齐（与内存分配策略相关）才能得新slice的容量，且新slice的容量要大于老slice容量1.25倍或者2倍。

下面简述主要的步骤：

  1. 当触发slice扩容时，调用growslice函数。
  2. 调用nextslicecap函数初步计算扩容大小，小切片增长2倍平滑过渡到大切片增长1.25倍，临界值是256。
  3. 根据内存分配策略，调用roundupsize函数进行内存对齐计算。
  4. 最后申请内存，gc着色，手动清理，并拷贝老slice的数据到新内存中。

4. 切片作为函数参数会被改变吗？

切片slice本质是一个结构体，所以在作为函数参数时，主要看怎么传参。

如果直接传slice，那么实参是形参的一个拷贝，实参并不会被函数中的形参的操作改变；

如果传的是slice指针，那么就会影响到实参。

然而，无论传的是slice还是slice指针，因为slice底层数组在slice结构体中是一个指针，所以只要对底层数组进行操作，都将改变底层数组的数据，虽然不会改变slice结构体本身。

Go语言中的函数参数传递，只有值传递，没有引用传递。

所以通过s[i]=10这种操作或通过slice的array字段就能改变slice底层数组元素值。

5. 内建函数make和new的区别是什么？

make和new都是Go语言用来分配内存的内置函数。

但适用类型不同：

  • make：slice、map、channel等引用类型，返回一个值。
  • new：int、数组、结构体等值类型，返回一个指针。

make返回初始化之后的类型的引用。

new会为类型的新值分配已置零的内存空间，并返回指针。

散列表map

1. map是什么？

维基百科中这样定义map：在计算机科学里，被称为相关数组、map、符号表或者字典，它是由一组<key，value>对组成的抽象数据结构，并且同一个key只会出现一次。

map的相关操作：增删改查

关键点：由组成；key是唯一的。

map的设计也被称为The dictionary problem，设计目的：

设计一种数据结构用来维护一个集合的数据，并且可以同时对集合进行增删改查的操作。

这种数据结构一般有两种：哈希查找表（Hash table）、搜索树（Search tree）

哈希查找表：用一个哈希函数将key分配到不同的桶（bucket)；性能高，开销主要在于哈希函数的计算以及数组的常数访问时间。

缺点：会产生哈希碰撞/冲突，即不同的key被哈希到同一个桶。

解决哈希碰撞：链表法、开放地址法

链表法：将一个桶实现成链表，落在同一个桶中的key都会插入这个链表。

开放地址法：在碰撞发生之后，根据一定的规律，在桶的后面挑选空位，用来放置新的key。

搜索树：一般采用自平衡搜索树，如：AVL树、红黑树等。

自平衡搜索树法的最差搜索效率是**O(logN)，而哈希查找表是O(N)**。

哈希查找表的平均查找效率是**O(1)**。

两者还有一个区别：

遍历自平衡搜索树，返回key的序列，一般按从小到大的顺序；

哈希查找表则是乱序的。

2. map的底层实现原理是什么？

Go语言中map是采用哈希查找表，并使用链表法解决哈希冲突。

go底层源码中，map的结构体是hmap（hashmap）：

bmap就是桶bucket，一个桶只能装8个<key,value>键值对。

当map的key和value都不是指针时，并且size < 128字节，bmap标记为不含指针，这样可以避免GC时扫描整个hmap，提升效率。

但实际上，bmap还有一个overflow的指针类型字段，这里巧妙得将overflow移动到hmap的extra字段内。

当key/value都不含指针时，启用overflow和oldoverflow字段。

在程序启动时，Go会检测CPU是否支持aes，如果支持，则使用aes hash，否则使用memhash。

源码路径：src/runtime/alg.go的alginit()函数。

哈希函数分为加密型和非加密型。

2.4 key定位过程

map查找key，调用的mapaccess1函数。

  1. key 经过哈希计算得到哈希值，共64位。（仅讨论64位机）

  2. 使用初始化hmap时就赋值的B，取哈希值低B位，转成十进制，那么这个十进制数就是该key落在几号桶。

  3. 取哈希值的高8位，找到该key在bucket中的槽位。

    最开始因为bucket里没有key，在遍历完bucket中所有的槽位，包括overflow的槽位后，找不到相同的key时，该key就会被放到第一个槽位。

当两个不同的key落在同一个bucket中，也就是发生了哈希冲突。

使用链表法来解决冲突：在bucket中，从前往后找到第一个空位，放入新加入的有冲突的key。

上图中，初始化时B=5，那么bucket的数量就是2^5=32个。

  1. 首先计算key的哈希值。
  2. 使用低5位00110(低B位)，转换成十进制6，找到第6个bucket。
  3. 使用高8位01111011，转换成十进制123，在第6号bucket中查找tophash值为123的槽位，这里找到了tohash数组的2号槽位。
  4. 找到2号槽位后，那么对应的key应该落在keys[2]的key2，则对应的value就是value[2]的value2。

如果在bucket中没找到，并且overflow不为空，还要继续去overflow bucket中查找，直到找到或所有槽位都找遍了，仍然没有找到，此时返回key对应类型的零值。

源码目录：src/runtime/map.go的mapaccess1函数。

tophash是存储该bucket的key的tophash值以及表征该bucket的状态，以下是几个状态：

emptyRest = 0 // 空的cell，初始化时的状态，也表示该cell后续的所有cell都是空的，包括overflow bucket
emptyOne = 1 // 当前的cell是空的
evacuatedX = 2 // key,value已经搬迁完毕，但key都在新bucket的前半部分
evacuatedY = 3 // 同上，key在后半部分
evacuatedEmpty = 4 // 空的cell，且cell已经被搬迁到新的bucket
minTopHash = 5 // tophash的最小正常值

当 emptyOne < tophash[0] < minTopHash，即tophash[0]是evacuatedX、evacuatedY、evacuatedEmpty之一时，表示该bucket上的key已经被全部搬迁到新bucket。

在tophash数组中，为了和正常的哈希值（高8位）区分开，会将所有的key的哈希值加上一个增量minTopHash，这样就可以区分正常的tophash和表示搬迁状态的哈希值了。

2.5 map的赋值过程

向map插入或修改key，调用的是mapassign函数。

mapassign函数返回的指针就是指向该key所对应的value值所在的内存地址。

赋值流程如下：

  1. 检查map的标志位flag，是否存在并发写操作。如果存在，程序会直接panic。
  2. 计算key的哈希值，找到落入的bucket。
  3. 检查该bucket是否处于扩容状态。
  4. 如果处于扩容状态，那么进行迁移操作，将老bucket的key分散迁移到2个新bucket上，只有迁移完成后，才能安全地在新bucket里定位key要放置的地址。
  5. 定位key的放置位置，准备2个指针（inserti指向key的hash值在tophash数组所在位置，insertk指向key的位置)，循环遍历bucket的tophash数组（包括overflow bucket）。
  6. 在遍历过程中，如果有空闲的cell，inserti会指向第一个空的tophash槽位，insertk会指向第一个空闲的cell。
  7. 如果遍历完成后，没有找到key的存在，那么意味着插入新的key，而不是更新原有的key，那么最终key的安置位置就是第一次发现的空闲的cell。
  8. 如果没有找到key且该bucket的8个key都放置满了，在跳出循环后，inserti和insertk都为空，那么就需要在最后遍历的bucket后面挂上overflow bucket。当然，这里要检查map的状态，是否需要进行扩容，如果满足扩容条件，主动触发扩容操作，并在扩容完成，重新走一遍查找定位key的过程。
  9. 定位到key之后，计算出value的位置，即key的位置跨过8个长度即可，赋值即可。
  10. 更新map相关值，如果是插入新key，map的字段count值加1；并且将hashWriting写标志位清零。

那么，如果先后插入两个不同的key，刚好它们的低B位和高8位又刚好相同，map是如何存储这个两个键值对呢？

低B位相同表示都落在同一个bucket；

高8位相同表示它们的tophash相同。

在map的赋值流程中，遍历bucket时，会用一个指针记录第一个空闲的tophash槽位。当插入第一个key时，第一个空闲的槽位被占用了；当插入第二个key时，遍历桶时，发现找到的槽位已被占用，会判断第二个key与改槽位的key是否相同，如果相同则是更新操作，否则会继续查找：

  • 直到找到相同的key，进行更新操作。
  • 找到空闲的tophash槽位，并找不到相同key，进行增加操作。
  • 找不到相同的key，找不到空闲的tophash槽位(即8个槽位已满)，在overflow指针挂载新的overflowbucket，并把key存储载第一个槽位。当然会触发检查扩容条件，并决定是否扩容，如果是，则等待扩容完成后，重新定位key，并执行更新/增加操作。

当然，在查找操作时，即使tophash相同也会判断key是否相同。

2.6 map的删除过程

删除操作底层的执行函数是mapdelete。

删除过程如下：

  1. 检查是否存在并发操作。
  2. 计算key的哈希值，找到落入的bucket。
  3. 设置写标志。
  4. 检查此map是否正在扩容的过程中，如果是则直接触发一次搬迁操作。
  5. 两层循环，核心是找到key的具体位置，定位过程类型，即在bucket中挨个cell寻找。
  6. 找到对应位置后，对key或value进行清零操作。
  7. 将map中的count减1，将对应位置的tophash值改成emptyOne。
  8. 最后，检测该槽位后面是否都是空，若是将tophash改成emptyRest。
  9. 若前一步成功，将此cell之前的tophash值为emptyOne的槽位都改成emptyRest。

2.7 map的扩容过程

哈希表的设计目的就是为了快速查找目标key，然而随着向map添加的key越来越多，key发生哈希碰撞的概率将越来越大。

理想情况：一个bucket只装一个key，能达到O(1)，但空间换时间代价太高了。

实际情况：一个bucket可以装载8个key，先定位到某个bucket，再定位到具体槽位，即时间换空间。

无论空间换时间或时间换空间，中间必须有个度，而这个度就是装载因子loadFactor。

count是map里的元素个数，2^B表示bucket的总数。

在插入新key时，如果符合以下两个条件，就会触发扩容：

  1. 装载因子超过阈值6.5。
  2. overflow的bucket数量过多：
    • 当B<15，也就是bucket总数2^B小于2^15时，overflow的bucket数量超过2^B;
    • 当B>=15，也就是bucket总数2^B大于等于2^15，overflow的bucket数量超过2^15。

具体解释以上两个扩容条件：

  1. 每个bucket有8个空位，在没有溢出的，且所有桶都装满情况下，装载因子为8。因此当装载因子超过6.5时，表示很多bucket都快满了，对map的操作效率会变低，应该进行扩容。
  2. 在装载因子比较小的时候，map的操作效率也很低。map的元素总数少，但bucket总数多。

造成条件2的原因在于：不断地插入、删除元素。

以上两个条件扩容的策略并不相同：

  1. 条件1，元素多，bucket少。策略：将B加上1，bucket总数2^B直接变成原来的2倍。
  2. 条件2，元素少，bucket多。策略：开辟一个新的bucket空间，将老bucket中的元素移动到新bucket，使得同一个bucket的key排列地更加紧密。 即在overflow bucket的元素移动到新bucket，以节省空间，提高bucket利用率，map的操作效率。bucket的数量和之前一样，并不包括overflow bucket。

针对条件2的解决方案，有一种极端情况：如果插入的key的哈希值都一样，就会落到同一个bucket里，超过8个就会产生overflow bucket，结果也会造成overflow bucket过多。

此时，移动元素无法解决问题，整个哈希表已经退化成了一个链表，操作效率变成了O(n)。

map的扩容采取了一种称为“渐进式”的方式，原有的key不会一次性搬迁完毕，而是每次最多只搬迁2个bucket。

在满足条件1、2时，会先调用hashGrow函数。

实际上，hashGrow()函数并没有真正地进行搬迁，只是分配好新的bucket，并将老buckets加载到oldbuckets字段上。

真正进行搬迁bucket是在growWork()函数上，而调用growWork()函数是在mapassign和mapdelete函数中。

在插入、修改、删除key的时候，都会先检查oldbuckets是否搬迁完毕，即检查oldbuckets是否为nil，再尝试进行搬迁buckets的工作。

具体搬迁流程：

  1. 插入、修改、删除key时，检查oldbuckets是否为nil，非nil则进行下一步搬迁工作。
  2. 调用hashGrow()函数，申请新bucket的空间，处理相关标志位。
    • 如果是等量扩容，设置h.flags为sameSizeGrow。
    • 标志nevacuate被赋值为0，表示当前搬迁进度为0。
  3. 调用evacuate()函数进行搬迁，注意evacuate()每次只搬迁一个bucket，当然包括其overflow bucket。
  4. 同样需要两层循环，外层遍历bucket及其overflow bucket，内层遍历bucket的所有cell。
  5. 如果符合条件1，新的buckets数量是之前的2倍；如果符合条件2，新的buckets数量和之前相同。
  6. 对于条件2（等量扩容），从老buckets搬迁到新的buckets上，按照序号进行搬迁，即0号的buckets仍然搬迁到0号的buckets。
  7. 对于条件1，要重新计算key的哈希值，因为B+1，所以要看低B+1位，进行确定key落在哪个bucket。
  8. 确定要搬迁到的目标bucket后，首先将源key/value值复制到目的地相应的位置。
  9. 然后设置key在原始buckets的tophash为evacuatedX或是evacuatedY，表示搬迁到了新map的x part或y part。
  10. 最后设置新map的tophash值，key哈希值的高8位。

2.8 map的遍历过程

map的遍历过程看起来好像比较简单：遍历所有的bucket及其后面挂的overflow bucket，同时遍历每个bucket中的所有cell即可。

其实不然，还记得map的扩容不是一个原子操作，每次最多搬运2个bucket。

所以如果触发了扩容操作，那么map中的有些bucket已经搬迁了，有些还在没搬迁。

因此，map的遍历如果发生在扩容过程中，那么就需要遍历新老bucket。

在底层源码中，底层函数调用关系：先调用mapiterinit函数初始化迭代器hiter，然后循环调用mapiternext函数进行map遍历。

在mapiterinit中，随机确定一个初始遍历的bucket(it.startBucket)，并随机确定从哪个cell开始遍历。

那么，mapiternext函数就会从it.startBucket的第it.offset号的cell开始遍历，并从中取出key和value，直到又回到起点bucket，完成遍历过程。

只不过只用看它tophash的最低位，如果是0，则分配到X part；如果是1，则分配到Y part。

结论：map遍历的核心在于2倍扩容时，老bucket会分配到2个新bucket中去。遍历操作会按照新bucket的序号顺序进行，遇到老bucket未搬迁时，则要在老bucket中找到将来要搬迁到新bucket的key。

3. map中的key为什么是无序的？

无序的原因有2个：

  1. map的扩容会导致key被搬迁。
  2. map的遍历过程中，每次遍历都会从一个随机序号的bucket开始，并且从该bucket的随机序号的cell开始遍历。

4. map是线程安全的吗？

map不是线程安全的。

在查找、赋值、遍历、删除过程中都会检查写标志（hmap.flags），一旦发现写标志位为1，则直接panic。

赋值和删除函数会检查完写标志是复位状态（为0）之后，先将写标志位置为1，才会进行接下来的操作。

5. float类型可以作为map的key吗？

可以，但当key为math.NaN()，会由于精度问题，导致一些奇怪的问题。

具体类型有：bool、数字类型、字符串、指针、通道、接口类型、结构体、只包含上述类型的数组。

7. 如何比较两个map是否相等？

两个map深度相等的条件如下：

  1. 都为nil。
  2. 非空、长度相等，指向同一个map实体对象。
  3. 对应的key指向的value也是深度相等。

三个条件任意一条满足，则认为两个map相等。

注意：直接使用map1 == map2是错误的且无法通过编译，==只能比较map是否为nil，而不能比较两个map是否相等。

只能通过遍历map的每个元素，比较元素是否都是深度相等。
